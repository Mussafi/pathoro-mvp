# Pathoro v0.5 — Real Opportunity Ingestion (Architecture Only)

This version does not add scraping, a database, or auth. It prepares the data model, the mock data, and an internal prototype workflow so that real opportunity ingestion can be built safely later, without redesigning the app around it.

**Product correction:** manually posting every event by hand does not scale — Pathoro needs a path to automated discovery. But "automated" does not mean "scrape Instagram and Facebook at scale." The sections below describe how discovery and extraction stay separate, and why Meta platforms are treated as a discovery signal, never a source of truth.

## Why Pathoro should not blindly scrape the whole internet

Pathoro's core promise is that opportunities are surfaced because they are *reachable and meaningful for a specific person's path* — not because they were the easiest thing to scrape. Blind, wide-net scraping causes real problems:

- **Trust collapse.** A single broken, closed, or fake listing erodes confidence in every other opportunity Pathoro shows, including the ones that are genuinely good.
- **Consent and terms of service.** Many platforms explicitly prohibit scraping in their terms, and some (social platforms especially) actively detect and block it. Ingesting from those sources without permission is a legal and platform-risk problem, not just a technical one.
- **Data quality.** Scraped listings are frequently stale, ambiguous about cost/location, or written for a totally different audience than "someone looking for their next reachable step." Pathoro's value is curation and fit, not volume.
- **Safety.** An opportunity surfaced to a vulnerable or exploring user (career change, health change, community-seeking) carries more responsibility than a generic search result. A bad recommendation here has real personal cost.

Because of this, v0.5 is architecture-first: build the model and the review workflow before building any ingestion.

## Scraping vs. opportunity ingestion

These are not the same thing, and Pathoro is built around the second one:

| Scraping | Opportunity ingestion |
|---|---|
| Extracts everything a page contains | Extracts a specific opportunity, deliberately, into a defined schema |
| No consent model assumed | Prioritizes sources that publish structured, public, permission-friendly event data |
| Optimizes for volume | Optimizes for fit — does this belong on a specific route, for a specific kind of person? |
| Goes live automatically | **Every imported opportunity is human-reviewed before it becomes live** |
| Treats all sources the same | Explicitly separates recommended sources from restricted/risky sources |

## Recommended first sources

These are structured, public-facing, and generally either explicitly designed for discovery or have clear terms for programmatic or manual access:

- Eventbrite (public event listings, has an API)
- Luma (public event listings, similar in spirit to Eventbrite)
- Organizer websites and booking pages (the organizer's own canonical page)
- Local library calendars
- Community center calendars
- Parks & recreation department pages
- Coworking / makerspace calendars
- University public event listings
- Local newsletters (e.g. neighborhood or interest-based newsletters that list events)
- Volunteer boards (e.g. local volunteer-matching organizations)
- Direct Pathoro submissions (a host, mentor, or organizer submits their own opportunity)

## Restricted / risky sources — not used as a source of truth

- **Instagram** — no direct scraping or bulk extraction. Terms of service explicitly restrict this, and content is often ephemeral/unstructured. It can be used only as a *discovery signal* (see below), never as the page Pathoro extracts structured details from.
- **Facebook Events** — same concern; also frequently requires a logged-in session to view, which changes the legal and technical picture entirely. Same discovery-signal-only treatment as Instagram.
- **Google Maps / Places** — usage of this data is governed by Google's API terms, which restrict caching, redistributing, and using the data outside of a Google Maps context. This needs a deliberate compliance review before any use, not an assumption of "public means usable." Not used even as a discovery signal in v0.5.

If a source isn't on the recommended list or explicitly described as a discovery signal below, the default answer is **not yet**, not **find a workaround**.

## Instagram as signal, not source of truth

Pathoro should not depend on scraping Instagram at scale, but that does not mean Instagram is ignored entirely. The distinction:

- **Public visibility does not automatically mean safe automated collection.** A post being visible to a logged-out viewer is not the same as it being intended for bulk, automated extraction. Instagram's terms restrict scraping regardless of whether the content is public.
- **Meta platforms are unstable and restrictive as a core data source.** Layouts change, access gets rate-limited or blocked, and there is no stable API for this use case. Building Pathoro's opportunity pipeline on top of that instability would make the whole system fragile.
- **So Pathoro should not depend on scraping Instagram or Facebook at scale, structurally or as a growth strategy.**
- **But Instagram can still help *discover* organizers.** A local ceramics studio or running club posting on Instagram is a legitimate signal that "this organizer exists and is active." Pathoro can use that signal to identify the organizer, then look for — or ask the organizer for — their canonical source page (a website, an Eventbrite page, a newsletter, a booking page) and extract from *that*, never from the Instagram post itself.

In short: Instagram can point Pathoro at an organizer. It should never be the page Pathoro reads opportunity details from.

## AI-assisted opportunity discovery

The target pipeline, once built, separates *finding* opportunities from *trusting* them:

1. **Discover local organizers and opportunity signals.** This can include public web search, community boards, newsletters, and yes, noticing active local organizers on social platforms.
2. **Prefer canonical source pages outside Meta.** For any organizer discovered, look for: their own website, Eventbrite, Luma, a newsletter, a booking page, or a library/community calendar listing. These are the pages Pathoro is actually allowed to read structured data from.
3. **Extract structured opportunity details from those canonical sources** — not from the Instagram or Facebook post that led there.
4. **Use Instagram/Facebook only as discovery clues where appropriate** — never as the extraction target itself.
5. **Create draft opportunities** with `status: "needs_review"`, never `"live"`.
6. **Human or organizer review before publishing.** A person (Pathoro team member or the organizer themselves) confirms the details are accurate before anything reaches `"live"`.

## Automated discovery without manual posting

To scale past manual, one-by-one posting without resorting to bulk Meta scraping, discovery can combine several lower-risk channels:

- **City-based source discovery** — for a given city, build a list of the canonical calendar/listing pages (library, parks & rec, community centers, coworking spaces, university event pages) once, then check them on a schedule.
- **Organizer watchlists** — once an organizer is identified (via any channel, including social discovery), watch their *canonical* page (website, Eventbrite, newsletter) for new listings, not their social feed.
- **Event/calendar crawling from permitted public pages** — pages that are explicitly public listings (not walled behind login), respecting robots.txt and terms of service.
- **APIs where available** — Eventbrite and Luma both have APIs; prefer these over any form of HTML scraping wherever they exist.
- **Newsletter / RSS ingestion** — many local newsletters and calendars publish RSS feeds, which are explicitly meant for programmatic consumption.
- **User-submitted links** — a Pathoro user pastes a link to something they found; Pathoro extracts from that specific page with the user's implicit request.
- **Organizer-submitted opportunities** — the most direct and lowest-risk channel: the organizer tells Pathoro about their own opportunity directly.

Every channel above still ends at the same place: a draft with `status: "needs_review"`, reviewed by a human before it can become `"live"`.

## Review requirement

**Every imported opportunity is human-reviewed before it becomes live.** No ingestion path — automated or manual — skips this. This is enforced by the `status` field in the data model: nothing reaches `"live"` without passing through `"needs_review"` first, and that transition is a deliberate human action, not an automatic one.

## What v0.5 actually builds

1. A structured `Opportunity` data model ([`lib/opportunitySchema.ts`](../lib/opportunitySchema.ts)) that has room for source attribution, trust signals, and review status — even though nothing is ingested from the web yet.
2. Updated mock seed data ([`lib/opportunities.ts`](../lib/opportunities.ts)) using that schema, clearly marked `sourceType: "mock_seed"` and `status: "preview"`. Nothing here is claimed to be a real, imported opportunity.
3. An internal-only prototype page at `/admin/opportunity-ingestion` that walks through what the future review workflow will feel like: paste a source URL (including, per the discovery model above, an Instagram/Facebook link that would need to resolve to a canonical page), pick a source type, get a mock "extracted" preview, edit it, see a route-fit suggestion, and see (but not actually trigger) an approval step. This page is not linked from the public app and does not call any real extraction service — it is a UI prototype of the workflow, not a working pipeline.
4. `/route-planning` now reads opportunity cards from the structured model instead of the old ad hoc shape, with visible "Preview data · Mock seed · Real ingestion coming soon" labeling so nothing on the public-facing app is mistaken for a real, live opportunity.

## What is intentionally still missing

- No real scraping or extraction (the "Extract opportunity" button on the admin prototype returns canned mock data, not a real parse of the pasted URL)
- No automated discovery pipeline — city-based source discovery, organizer watchlists, calendar crawling, and newsletter/RSS ingestion are all described above as target architecture, not implemented
- No canonical-source resolution — there is no logic yet that takes an Instagram/Facebook link and finds the organizer's real source page
- No database — everything still lives in `lib/opportunities.ts` and resets on reload
- No auth — the `/admin/opportunity-ingestion` route is reachable by anyone who knows the URL; it is not protected, and it is not linked publicly
- No real approval persistence — the "Approve for map" action in the prototype is mock-only and does not change what appears on `/route-planning`
