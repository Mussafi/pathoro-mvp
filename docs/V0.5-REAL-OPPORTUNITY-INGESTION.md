# Pathoro v0.5 — Real Opportunity Ingestion (Architecture Only)

This version does not add scraping, a database, or auth. It prepares the data model, the mock data, and an internal prototype workflow so that real opportunity ingestion can be built safely later, without redesigning the app around it.

## Why Pathoro should not blindly scrape the whole internet

Pathoro's core promise is that opportunities are surfaced because they are *reachable and meaningful for a specific person's path* — not because they were the easiest thing to scrape. Blind, wide-net scraping causes real problems:

- **Trust collapse.** A single broken, closed, or fake listing erodes confidence in every other opportunity Pathoro shows, including the ones that are genuinely good.
- **Consent and terms of service.** Many platforms explicitly prohibit scraping in their terms, and some (social platforms especially) actively detect and block it. Ingesting from those sources without permission is a legal and platform-risk problem, not just a technical one.
- **Data quality.** Scraped listings are frequently stale, ambiguous about cost/location, or written for a totally different audience than "someone looking for their next reachable step." Pathoro's value is curation and fit, not volume.
- **Safety.** An opportunity surfaced to a vulnerable or exploring user (career change, health change, community-seeking) carries more responsibility than a generic search result. A bad recommendation here has real personal cost.

Because of this, v0.5 is architecture-first: build the model and the review workflow before building any ingestion.

## Scraping vs. opportunity ingestion

These are not the same thing, and Pathoro is built around the second one:

| Scraping | Opportunity ingestion |
|---|---|
| Extracts everything a page contains | Extracts a specific opportunity, deliberately, into a defined schema |
| No consent model assumed | Prioritizes sources that publish structured, public, permission-friendly event data |
| Optimizes for volume | Optimizes for fit — does this belong on a specific route, for a specific kind of person? |
| Goes live automatically | **Every imported opportunity is human-reviewed before it becomes live** |
| Treats all sources the same | Explicitly separates recommended sources from restricted/risky sources |

## Recommended first sources

These are structured, public-facing, and generally either explicitly designed for discovery or have clear terms for programmatic or manual access:

- Eventbrite (public event listings, has an API)
- Local library calendars
- Community center calendars
- Parks & recreation department pages
- Coworking / makerspace calendars
- University public event listings
- Local newsletters (e.g. neighborhood or interest-based newsletters that list events)
- Volunteer boards (e.g. local volunteer-matching organizations)
- Direct Pathoro submissions (a host, mentor, or organizer submits their own opportunity)

## Restricted / risky sources — not used in v0.5 or until explicitly revisited

- **Instagram** — no scraping; terms of service explicitly restrict this, and content is often ephemeral/unstructured.
- **Facebook Events** — same concern; also frequently requires a logged-in session to view, which changes the legal and technical picture entirely.
- **Google Maps / Places** — usage of this data is governed by Google's API terms, which restrict caching, redistributing, and using the data outside of a Google Maps context. This needs a deliberate compliance review before any use, not an assumption of "public means usable."

If a source isn't on the recommended list, the default answer is **not yet**, not **find a workaround**.

## Review requirement

**Every imported opportunity is human-reviewed before it becomes live.** No ingestion path — automated or manual — skips this. This is enforced by the `status` field in the data model: nothing reaches `"live"` without passing through `"needs_review"` first, and that transition is a deliberate human action, not an automatic one.

## What v0.5 actually builds

1. A structured `Opportunity` data model ([`lib/opportunitySchema.ts`](../lib/opportunitySchema.ts)) that has room for source attribution, trust signals, and review status — even though nothing is ingested from the web yet.
2. Updated mock seed data ([`lib/opportunities.ts`](../lib/opportunities.ts)) using that schema, clearly marked `sourceType: "mock_seed"` and `status: "preview"`. Nothing here is claimed to be a real, imported opportunity.
3. An internal-only prototype page at `/admin/opportunity-ingestion` that walks through what the future review workflow will feel like: paste a source URL, pick a source type, get a mock "extracted" preview, edit it, see a route-fit suggestion, and see (but not actually trigger) an approval step. This page is not linked from the public app and does not call any real extraction service — it is a UI prototype of the workflow, not a working pipeline.
4. `/route-planning` now reads opportunity cards from the structured model instead of the old ad hoc shape, with visible "Preview data · Mock seed · Real ingestion coming soon" labeling so nothing on the public-facing app is mistaken for a real, live opportunity.

## What is intentionally still missing

- No real scraping or extraction (the "Extract opportunity" button on the admin prototype returns canned mock data, not a real parse of the pasted URL)
- No database — everything still lives in `lib/opportunities.ts` and resets on reload
- No auth — the `/admin/opportunity-ingestion` route is reachable by anyone who knows the URL; it is not protected, and it is not linked publicly
- No real approval persistence — the "Approve for map" action in the prototype is mock-only and does not change what appears on `/route-planning`
