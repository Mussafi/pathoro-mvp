# Pathoro v0.9 — Discovery Sources: Reddit Local Signal Layer (Plan Only)

This is a planning document. **No Reddit ingestion code, no Reddit API
keys, and no subreddit crawling in this version.** The goal is to extend
Pathoro's discovery model (established in
[V0.5](V0.5-REAL-OPPORTUNITY-INGESTION.md)) to cover Reddit as a source of
local opportunity signals, and to note the schema additions that
[V0.8](V0.8-DATABASE-PLAN.md)'s `opportunities` table will need once this is
built.

## How this fits the existing discovery model

V0.5 already draws a hard line between **discovery** ("this organizer or
event exists") and **extraction** ("read structured details from a
canonical, permission-friendly page"), and treats Instagram the same way:
useful for noticing that something is happening locally, never the page
Pathoro reads details from. Reddit gets the identical treatment:

> **Reddit is a local opportunity signal layer, not a source of truth.**

Local subreddits (`r/nashville`, `r/Austin`, `r/jerseycity`, `r/AskNYC`,
etc.) are one of the highest-density places people already post about real
local events, classes, groups, and meetups — often before that event has
any polished web presence. That makes Reddit valuable for *finding*
opportunities. It does not make Reddit a place Pathoro treats as
ground truth, and it is explicitly **not** a target for unlimited or
continuous scraping.

## Policy constraints (non-negotiable)

- Use the **official Reddit API** (OAuth, a registered script/app,
  read-only scopes) or another explicitly permitted access method — no
  scraping that bypasses Reddit's terms of service.
- **Respect rate limits.** Watchlisted subreddits are polled periodically
  (e.g. a scheduled job checking recent posts), never continuously or in a
  tight loop.
- **Respect deletions and removals.** If a post or its linked content is
  deleted/removed by the time it's reviewed, the draft is discarded or
  flagged invalid — never published from cached/stale content.
- **No unnecessary personal data.** Do not store Reddit usernames, user
  IDs, comment authors, or any other personal data beyond what's needed to
  re-find the specific post being reviewed (the subreddit name and the post
  URL — see schema below). No profile scraping, no cross-referencing
  Reddit accounts with anything else.
- **Never publish automatically.** Every Reddit-derived draft lands at
  `status: needs_review`, with no exception — including when a clean
  canonical source link is found. A human always reviews before anything
  reaches `live`, exactly like every other ingestion path in v0.5/v0.6.

## Local subreddit watchlist (by city)

A maintained, explicit list — not an auto-discovered crawl of "all
subreddits." Example shape:

| City | Subreddits |
|---|---|
| Austin, TX | `r/Austin` |
| Nashville, TN | `r/nashville` |
| Jersey City, NJ | `r/jerseycity` |
| New York, NY | `r/AskNYC`, `r/nyc` |

This list grows one city at a time, by hand, the same way the "recommended
first sources" list in V0.5 does for calendars and event platforms. It is
config, not a discovered/crawled set of communities.

## Keyword filter

Posts in watchlisted subreddits are only worth looking at if they match
signal keywords in the title or body:

```
event, meetup, class, workshop, volunteer, beginner, group, running,
hiking, art, ceramics, career, networking, community, free,
open to beginners
```

This is a first-pass filter to cut noise before anything reaches a human —
not a guarantee of relevance. Keyword-matched posts still go through the
canonical-source and review steps below.

## Discovery pipeline (design, not built yet)

1. **Poll** each watchlisted subreddit's recent posts on a schedule (not
   continuously), via the official Reddit API, respecting rate limits.
2. **Filter** posts by the keyword list above (title + body).
3. **Look for a canonical source** in the post body, the post's link (if
   it's a link post), or top-level comments: organizer website, Eventbrite,
   Luma, a Google Form, a library/community calendar, or a booking page —
   the same "recommended first sources" list from V0.5.
4. **If a canonical source is found:** create a draft by running the
   existing v0.6 extraction pipeline (`/api/ingest-opportunity`) against
   *that* canonical URL, not the Reddit post. The draft's `sourceType`
   reflects the canonical page (e.g. `eventbrite`, `direct_submission`), and
   is additionally tagged with where it was *discovered* (see schema below:
   `discovered_from_platform: "reddit"`, `discovered_from_url`, `subreddit`).
   Status is still `needs_review` — discovery method never shortcuts
   review.
5. **If no canonical source is found:** create a draft directly from the
   Reddit post's own content (title, body text, any date/location
   mentioned). `sourceType: "reddit_signal"`, `status: "needs_review"`,
   `canonical_source_found: false`. This draft leans more heavily on human
   judgment during review, since there's no organizer-controlled page to
   verify against.
6. **Human review, always.** Nothing from this pipeline ever reaches
   `live` without going through the same admin review flow as every other
   ingestion path.

## New `sourceType` values

- **`reddit_signal`** — the opportunity's source is the Reddit post itself;
  no canonical page was found. Added to `OpportunitySourceType` in
  `lib/opportunitySchema.ts` alongside the existing values, when this is
  built.
- **`subreddit_watchlist`** — requested for completeness, but with a design
  note: this isn't really a *source type for an opportunity* (a watchlist
  entry isn't itself an opportunity someone can approve). It more naturally
  belongs as its own small config concept — either a static list (like the
  table above) or, later, a lightweight `subreddit_watchlists` table
  (`city`, `subreddit`, `active`) if the list grows enough to want DB-backed
  management instead of a hardcoded list. Not added to the `Opportunity`
  type's `sourceType` union for that reason; kept here as a named concept
  for whichever config shape is chosen when this is built.

## Optional `opportunities` table fields (extends V0.8's schema)

These are additive to the schema in
[V0.8](V0.8-DATABASE-PLAN.md#schema) — nullable, and only populated for
opportunities that came through a discovery pipeline (Reddit or, later,
other signal sources):

| Column | Type | Notes |
|---|---|---|
| `discovered_from_platform` | `text` | e.g. `"reddit"`. Nullable — unset for directly-submitted or manually-entered opportunities. Generalized beyond just Reddit so future signal sources (other forums, local Slack/Discord communities, etc.) reuse the same column. |
| `discovered_from_url` | `text` | The Reddit post URL (or future equivalent) the signal came from. This is the minimum needed to re-find/re-verify the source — not a general activity log. |
| `subreddit` | `text` | e.g. `"Austin"`. Nullable, Reddit-specific. |
| `canonical_source_url` | `text` | The organizer site / Eventbrite / Luma / form / calendar / booking page found during discovery, if any. Distinct from `source_url` (which V0.8 already has) so "where we found the signal" and "the canonical page we actually extracted from" stay separate even when they end up being the same URL. |
| `canonical_source_found` | `boolean` | Whether step 3 of the pipeline found a canonical source. Drives whether `sourceType` is the canonical type or `reddit_signal`. |
| `discovery_confidence` | `text` (`low` / `medium` / `high`) | A rough triage signal for the admin review queue — e.g. `high` when a canonical source was found and the keyword match is strong, `low` when it's a vague keyword-only match on the Reddit post text alone. **Never used to auto-publish** — it only affects review ordering/prioritization. |
| `reviewed_by` | `text` | Nullable. As noted in V0.8, there's no individual admin identity yet under the shared-secret admin model — this column exists so no future migration is needed once real admin accounts exist, but it stays unpopulated until then. |
| `reviewed_at` | `timestamptz` | Already part of V0.8's schema; reused here, not duplicated. |

## Explicitly out of scope for v0.9 (this plan)

- Reddit API credentials/registration.
- Any actual polling job, cron, or crawler.
- Comment-thread parsing beyond looking for a canonical link.
- Reddit DMs/inbox, modmail, or any two-way interaction with Reddit.
- Karma/vote-based ranking or sentiment analysis of posts.
- Any form of automatic publishing — every Reddit-derived draft requires
  human review, with no exceptions carved out in this plan.
- Subreddits outside an explicitly maintained watchlist — no "search all of
  Reddit" mode.

## Implementation note (for whenever this is built)

Reddit discovery slots into the existing v0.6/v0.7/v0.8 pipeline as an
additional **draft producer**, not a parallel system: it still ends at the
same `IngestionDraft` shape feeding the same admin review UI and the same
`opportunities` table from V0.8. The only new work is (a) the
polling/filtering step that finds candidate posts, and (b) the
discovery-metadata columns above — everything downstream of "draft exists"
is unchanged.
