# Pathoro v0.8 — Database-Backed Reviewed Opportunities (Plan Only)

This is a planning document. No application code changes in this version — the
goal is to pick a backend before v0.8 implementation starts.

## Why v0.7 isn't enough

v0.7 made reviewed opportunities feel real (extract → edit → approve → show
on the route map → detail page), but everything lives in one browser's
`localStorage` (`pathoro.reviewedOpportunities.v1`). That means:

- Opportunities approved on one device don't show up anywhere else.
- Clearing browser data deletes them permanently.
- There is no single source of truth Pathoro (or a future teammate) can look
  at to see what's actually live.

v0.8 replaces that local store with a real database, without adding auth or
scraping — this is purely a persistence upgrade.

## Requirements

- Approved opportunities persist beyond `localStorage` (survive reloads,
  other devices, other browsers).
- `/route-planning` can read live reviewed opportunities from the database.
- `/admin/opportunity-ingestion` can write reviewed opportunities to the
  database.
- No public auth yet.
- A simple admin-only protection mechanism should be easy to add *later*
  without redoing the database choice.
- Easy to deploy on Vercel (this app deploys there).
- Minimal added complexity — Pathoro is still an MVP.

## Options considered

## Recommendation: Vercel Postgres (Neon)

For where Pathoro is right now, **Vercel Postgres (Neon)** is the best fit.

- It's a first-party Vercel integration: adding it from the Vercel dashboard
  auto-injects the connection string into both Preview and Production
  environments — no separate account, no manual secret wiring.
- The `Opportunity` type in `lib/opportunitySchema.ts` is already a flat,
  relational shape. A single `opportunities` table maps onto it directly —
  no translation layer, no NoSQL modeling decisions.
- It's plain SQL. That keeps v0.8 small: one query helper module, a couple
  of parameterized queries, no new client-side SDK, no new concepts (like
  Supabase's Row Level Security) to get right on day one.
- Admin-only protection is explicitly **decoupled from the database choice**
  in this plan (see below), so Postgres's lack of bundled auth isn't a
  downside for this stage.
- Neon's own web console gives basic table/SQL visibility if we ever need to
  eyeball the data, without adopting a second product surface.

### Why not the others (for right now)

- **Supabase** is a strong second choice and worth revisiting once Pathoro
  wants real user auth (Supabase Auth would then pull double duty for admin
  login and, later, end-user accounts). Today it would mean standing up a
  second vendor account and making Row Level Security decisions for
  functionality we can get for free by keeping all writes server-side.
- **Firebase/Firestore** is the weakest fit: the data is relational and
  already typed that way, Firestore's document model would require
  re-thinking the schema for no real benefit, and it has no native Vercel
  integration (manual service-account credential handling).
- **SQLite/Turso** is a reasonable minimal option and would work fine at
  this scale, but it adds a less-established vendor/ecosystem for
  Next.js-on-Vercel with no advantage over Neon for a small relational
  table like this one.

## Admin-only protection (decoupled from the database)

Whichever database we use, "simple admin-only protection later" doesn't
need to come from the database vendor. The plan is:

- Keep `/admin/opportunity-ingestion` and any new write API routes
  unauthenticated for v0.8, same as today — internal-only, not linked
  publicly.
- When protection is wanted, add a single shared-secret check (an
  `ADMIN_TOKEN` environment variable, checked via Next.js middleware or a
  cookie set after entering the token) in front of `/admin/*` and the
  opportunity-write API route. This is a few lines of code and doesn't
  require picking an auth provider.
- If Pathoro later wants real accounts/roles, that's a separate, bigger
  decision (Supabase Auth, Clerk, NextAuth, etc.) independent of which
  database stores opportunities.

## Shape of the v0.8 implementation (for the next step, not built yet)

- **Schema**: one `opportunities` table mirroring the `Opportunity` type in
  `lib/opportunitySchema.ts` (same fields: `id`, `title`, `source_url`,
  `source_name`, `source_type`, `city`, `state`, `location_label`,
  `date_label`, `cost_label`, `host_name`, `description`, `route_id`,
  `opportunity_type`, `who_it_is_for`, `path_it_supports`,
  `what_it_may_open_next`, `effort_level`, `friction_level`, `trust_level`,
  `status`, `href`, plus `created_at`/`updated_at`).
- **`lib/db.ts`**: a small server-only Postgres client using
  `@vercel/postgres` (or `@neondatabase/serverless`).
- **`lib/opportunitiesDb.ts`**: server-only query functions —
  `getLiveOpportunities()`, `getAllReviewedOpportunities()`,
  `upsertOpportunity()` — replacing the localStorage functions in
  `lib/reviewedOpportunities.ts`.
- **`app/api/opportunities/route.ts`**: a new API route (same pattern as
  the existing `app/api/ingest-opportunity/route.ts`) — `POST` to
  approve/save an opportunity, `GET` for the admin page's "Locally
  approved opportunities" list (renamed to just "Approved opportunities"
  once it's real).
- **`/route-planning`**: becomes (or gains) a server-side data fetch — read
  live opportunities from the database at request time and pass them down
  to the existing client components, instead of the `useSyncExternalStore`
  localStorage hook. Keeps DB credentials server-side only.
- **`/admin/opportunity-ingestion`**: keeps its current client-side edit
  flow, but "Approve for map" calls the new API route instead of writing to
  `localStorage`.
- **Migration**: `localStorage` reviewed-opportunity code is retired, not
  kept as a fallback — one source of truth avoids confusing dual-write bugs.

## Explicitly out of scope for v0.8

- Public auth or end-user accounts.
- Real admin auth (only the env-var shared-secret placeholder described
  above, and only if/when we decide to add it).
- Any change to the ingestion/extraction logic in
  `app/api/ingest-opportunity/route.ts`.
- Realtime updates, caching layers, or an ORM — plain SQL is enough at this
  scale.
