# Pathoro v0.8 — Database-Backed Reviewed Opportunities (Plan Only)

This is a planning document. **No application code changes in this version.**
The goal is to pick a backend and design the schema/API/security/migration
plan before v0.8 implementation starts.

## Current state (end of v0.7)

- v0.6 can extract a draft opportunity from a real URL
  (`app/api/ingest-opportunity/route.ts`).
- v0.7 can approve that draft into `localStorage`
  (`lib/reviewedOpportunities.ts`, key `pathoro.reviewedOpportunities.v1`).
- `/route-planning` reads local reviewed opportunities and merges them with
  the mock seed data (`lib/opportunities.ts`).
- `/opportunity/[id]` can show a local reviewed opportunity's detail page.
- All of this is **browser-only** — nothing is shared across users, devices,
  or browsers, and clearing site data deletes it permanently.

## Goal for v0.8

Replace `localStorage`-backed reviewed opportunities with a real, persistent
database, without adding public auth, user accounts, organizer accounts,
automated crawling, or payments. Core flow after v0.8:

```
Admin pastes URL
  → extraction creates draft (unchanged, v0.6)
  → admin edits/reviews (unchanged UI, v0.7)
  → admin approves
  → opportunity saves to the database
  → /route-planning reads live opportunities from the database
  → /opportunity/[id] loads opportunity detail from the database
```

## Constraints

- Do not build code yet — this document is the plan.
- No full public auth, no user accounts, no organizer accounts yet.
- No automated crawling, no payments.
- Keep it simple — Pathoro is still an MVP.
- Must deploy easily on Vercel.
- Must work with the existing Next.js App Router structure.

## Options considered

Evaluated against the dimensions that actually matter for where Pathoro is
headed, not just where it is today:

| | Supabase Postgres | Vercel Postgres (Neon) | Firebase (Firestore) | SQLite/Turso |
|---|---|---|---|---|
| Ease of setup | Low-medium — new account, decide on Row Level Security up front | Lowest — add from Vercel dashboard, get a connection string, write SQL | Medium — separate Google Cloud project, service-account JSON handling | Low — add from Vercel dashboard, or a local file in dev |
| Vercel compatibility | Marketplace integration, env vars auto-added | Native/first-party integration, env vars auto-added to Preview + Production | No native integration — manual env var setup | Marketplace integration, env vars auto-added |
| Simple admin write flow | Straightforward — server-side service-role key bypasses RLS for writes | Straightforward — plain parameterized SQL from an API route | Straightforward via `firebase-admin`, but a different SDK/mental model | Straightforward — plain parameterized SQL |
| Public read flow | RLS policy restricts anon reads to `status = 'live'` directly at the DB layer | Enforced only in application code (the query itself must filter `status = 'live'`) | Firestore security rules can restrict reads, but rules are a separate DSL to learn | Enforced only in application code |
| Future auth / admin roles | **Supabase Auth is bundled** — admin roles, multiple admins, and row-level permissions grow on the same Postgres instance | None bundled — would need a separate provider (Clerk/NextAuth/Auth.js) added later | Firebase Auth is bundled, but ties auth to a different vendor than the data | None bundled |
| Future organizer submissions | Natural fit — Supabase Auth + RLS lets an organizer's account write only their own rows | Possible, but requires bolting on an auth provider first, then wiring its user IDs into Postgres permissions by hand | Possible with Firebase Auth + Firestore rules, but the data model split (relational schema vs. document rules) adds friction | Possible, same caveat as Neon |
| Future geolocation/search | Postgres extensions available (PostGIS for geo, `pg_trgm`/`pgvector` for text/semantic search) on the same database | Same Postgres engine, same extensions generally available — comparable to Supabase here | Firestore has weak native geo-query support (geohash workarounds) and no full-text/semantic search built in | SQLite/libSQL has limited extension support compared to Postgres; geo/search would need external tooling |
| Long-term flexibility | High — one Postgres instance can grow into auth, storage, geo, and search without changing vendors | High for the database itself, but auth/storage/search would each be a separate vendor decision later | Lower — NoSQL schema decisions made now are harder to unwind, and geo/search need bolt-ons regardless | Medium — fine at small scale, but the ecosystem thins out as needs grow |
| Risk / complexity (right now) | Slightly higher day-one complexity (must configure RLS correctly, or data leaks) | Lowest day-one complexity (no RLS concept at all) | Medium — new query paradigm, manual credential wiring on Vercel | Low, but smaller ecosystem/community means less prior art to lean on |
| Free tier | Generous | Generous | Generous | Generous |

## Recommendation: Supabase Postgres

**Supabase Postgres** is the recommended choice for Pathoro right now.

This is a change from treating "simplest possible for v0.8 alone" as the only
criterion. Weighed against where the constraints say Pathoro is *headed* —
admin roles, organizer submissions, and geolocation/search are all named as
explicit future needs — Supabase wins on long-term flexibility without
costing much extra complexity today:

- It's still plain Postgres underneath, so the schema below (a flat
  `opportunities` table) applies identically to Supabase or Neon — no
  NoSQL modeling tradeoff.
- **Auth is bundled and grows with the roadmap.** Admin roles and, later,
  organizer accounts are the *same* Supabase Auth + Row Level Security
  system, just with more policies added over time. Choosing Neon now would
  mean bolting on a separate auth vendor (Clerk/NextAuth) the moment admin
  roles or organizer accounts become real — a second migration, not a
  continuation.
- **Geolocation and search extensions live on the same database** (PostGIS,
  `pg_trgm`, `pgvector`), so "opportunities near me" or better matching
  later doesn't require a new service.
- The one real cost is Row Level Security: it must be configured correctly
  from day one (see Security, below) so public reads can't leak
  non-`live` rows. This is a one-time, well-documented setup, not ongoing
  complexity.
- Supabase Studio gives a table editor and SQL editor for free, useful for
  eyeballing/fixing data by hand during the MVP stage.

### Why not the others

- **Vercel Postgres (Neon)** is genuinely the lowest-friction option for
  v0.8 in isolation — first-party Vercel integration, zero extra concepts.
  It would be the right call if the roadmap stopped at "one admin, no
  organizers, no geo/search." Since it doesn't, Neon would likely mean
  re-platforming onto Supabase (or a separate auth vendor) later instead of
  extending what's already there. Worth reconsidering only if the org/auth
  roadmap items get dropped.
- **Firebase/Firestore** is the weakest fit: the `Opportunity` data is
  already a flat, relational, typed shape — Firestore's document model adds
  a translation layer for no benefit, its geo-query support is weak
  (geohash workarounds, no native PostGIS-equivalent), and it has no native
  Vercel integration (manual service-account credential handling).
- **SQLite/Turso** is fine at small scale and simple to set up, but has the
  thinnest extension ecosystem of the four for the geo/search future this
  plan explicitly anticipates, and a smaller Next.js/Vercel community to
  draw on than Postgres.

Indexes:
- `create index on opportunities (status)` — the public read path always
  filters on this.
- `create index on opportunities (route_id, status)` — matches the
  `/route-planning` query shape (route + live-first).

Note: the existing `OpportunityStatus` type is `draft | needs_review | live
| expired | preview`. This plan adds **`rejected`** so an admin can
explicitly decline a draft without deleting it (keeps extraction history and
avoids silently re-surfacing a rejected URL). `lib/opportunitySchema.ts`
gains `"rejected"` as a valid status when this is implemented.

### Optional: `ingestion_logs` (deferred, not required for v0.8)

Would record every extraction attempt (`source_url`, timestamp, success/
error, warnings, raw draft snapshot) for debugging and future dedupe
("have we already tried this URL?"). Not required for the core v0.8 flow —
the extraction API already surfaces warnings directly in the admin UI. Worth
adding once ingestion volume makes "did we already try this?" a real
question. **Not built in v0.8.**

### Not needed: `opportunity_reviews`

An audit trail of *who* reviewed/approved what only matters once there is
more than one admin identity. v0.8 has a single shared admin secret, not
individual admin accounts, so there's no "who" to attribute a review to yet.
Revisit this table when admin roles (plural) are added.

## API plan

| Route | Method | Access | Purpose |
|---|---|---|---|
| `/api/ingest-opportunity` | `POST` | Admin-only (see Security) | Unchanged from v0.6 — extracts a draft from a URL. Gets the same admin protection as the rest of `/admin/*` once that's added, but its logic doesn't change. |
| `/api/opportunities` | `POST` | Admin-only | Save a reviewed opportunity. Upserts by `id` (matches v0.7's `approvedId` reuse behavior — the client keeps sending the same `id` on repeat "Approve for map" clicks, so this must be `INSERT ... ON CONFLICT (id) DO UPDATE`, not a blind insert). |
| `/api/opportunities` | `GET` | Public | Reads. Query params `routeId` and `city` narrow the result, matching `filterForRoute()`'s current shape. **The `status = 'live'` filter is forced server-side and is never taken from the client**, so no query-string trick can expose non-live rows. Used by `/route-planning`'s server-side data fetch. |
| `/api/opportunities/[id]` | `GET` | Public | Loads one opportunity for `/opportunity/[id]`. Only returns rows where `status = 'live'`; anything else responds as not-found, which the page already renders as "saved locally in another browser/session or is no longer available" (copy can stay as-is or be softened slightly). |
| `/api/opportunities/[id]` | `PATCH` | Admin-only | **Deferred, not built in v0.8.** The `POST /api/opportunities` upsert-by-`id` already covers the main edit path (re-approve with changes). A dedicated `PATCH` (e.g. for admin-only status changes like `expired`/`rejected` without resending the whole draft) is a small, easy addition once there's a reason to build it — noted here so the route table is complete, but out of scope for v0.8's initial build. |
| Admin listing (either `GET /api/opportunities` with an admin key that lifts the `status = 'live'` filter, or a separate `GET /api/opportunities/admin`) | `GET` | Admin-only | Powers the "Locally approved opportunities" list on `/admin/opportunity-ingestion` (renamed once it's DB-backed — e.g. "Approved opportunities"), returning rows of every status, not just `live`. |

## Security plan

- **Public reads are read-only and status-gated.** No public route ever
  returns a row where `status != 'live'`. This is enforced twice, in
  layers:
  1. In the database itself, via a Supabase RLS policy on `opportunities`:
     `anon` role may `select` only where `status = 'live'`. Writes are
     **not** granted to `anon` at all.
  2. In the API route's query, as defense in depth, even though RLS is the
     real backstop.
- **Admin writes go through the Supabase service-role key**, used only in
  server-side API routes (`app/api/opportunities/route.ts`, etc.), never
  exposed to the browser. The service role bypasses RLS, so it can insert/
  update rows of any status.
- **Admin-only protection for v0.8 (temporary, explicitly not real auth):**
  a single shared secret — an `ADMIN_TOKEN` (or `ADMIN_PASSWORD`) environment
  variable — checked in Next.js middleware guarding `/admin/*` (the page)
  and the admin-only API routes (`POST /api/opportunities`, the admin
  listing `GET`, `/api/ingest-opportunity`). Simplest implementation: HTTP
  Basic Auth in `middleware.ts` comparing the supplied password to
  `ADMIN_TOKEN` — no login UI to build, no session/cookie logic, one env var
  per environment (local/Preview/Production, set to different values).
- **Why this is temporary:** it's one shared secret, not an identity. There's
  no way to tell which admin did something, no per-admin permissions, no
  revocation short of rotating the one shared value, and no audit trail
  (hence no `opportunity_reviews` table yet — there's no "who" to log). It
  is a placeholder that stops the admin page from being wide open on the
  public internet, nothing more.
- **What replaces it later:** Supabase Auth, once there's a reason to have
  more than one admin identity or to support organizer logins. Because the
  data already lives in Supabase, this is additive — swap the middleware
  check for a Supabase session check and add RLS policies keyed to
  `auth.uid()` / a `role` claim, rather than re-platforming.

## Migration plan

- **Ordering on `/route-planning`:** live database opportunities take
  priority over mock seed opportunities for a given route. Today's
  `mergeWithSeed()` puts reviewed opportunities first and fills in with
  `routeOpportunities` (the seed data) after; the same ordering carries
  over, just with "reviewed" now meaning "fetched from the database with
  `status = 'live'`" instead of "read from `localStorage`." Routes with no
  live DB opportunities yet still show seed data underneath, so the map
  never looks empty.
- **`/opportunity/[id]` lookup order:** database (`status = 'live'`) → seed
  data (`lib/opportunities.ts`) → not-found. No `localStorage` lookup in
  production.
- **Plant-Based Cooking Class must keep working.** It's a `mock_seed` item
  in `lib/opportunities.ts` with its own static page and an explicit
  `href`. Nothing in this plan touches that file or that route — it's
  preserved by construction, not by special-casing.
- **`localStorage` reviewed-opportunity code** (`lib/reviewedOpportunities.ts`,
  `lib/useReviewedOpportunities.ts`) is retired from the production path.
  Whether to keep it as a **dev-only fallback** (e.g. active only when no
  database connection string is configured, so contributors can run the app
  locally without provisioning Supabase) is an open call — leaning toward
  keeping it behind an explicit `if (!hasDatabaseUrl)` guard rather than
  deleting outright, since it's already-working code that costs nothing to
  keep gated off. Final call deferred to implementation time.

## Implementation steps (for the next version — not built now)

1. Provision a Supabase project; add the Supabase Vercel integration so env
   vars land in Preview + Production automatically.
2. Write the `opportunities` table migration (SQL): columns above, indexes,
   RLS enabled with the public-read-live policy, no policies granting
   `anon` write access.
3. Add `lib/db.ts` (Supabase server client, service-role key, server-only)
   and `lib/opportunitiesDb.ts` (typed query functions: `getLiveOpportunities`,
   `getOpportunityById`, `getAllOpportunitiesForAdmin`, `upsertOpportunity`).
4. Add `middleware.ts` with the shared-secret Basic Auth check for `/admin/*`
   and the admin-only API routes.
5. Add `app/api/opportunities/route.ts` (`GET` public list with a
   server-forced `status = 'live'` filter, `POST` admin-protected upsert).
6. Add `app/api/opportunities/[id]/route.ts` (`GET` public single,
   `status = 'live'` only).
7. Update `/route-planning` to fetch live opportunities server-side (an
   async Server Component data fetch, or a server-side call inside the
   existing page) and pass them down to the current client component tree,
   replacing the `useReviewedOpportunities` localStorage read on the public
   path.
8. Update `/opportunity/[id]` to fetch from the database first (server-side,
   `status = 'live'` only), falling back to seed data, replacing the
   localStorage lookup in production.
9. Update `/admin/opportunity-ingestion`'s "Approve for map" to `POST` to
   `/api/opportunities` instead of writing to `localStorage`; update the
   "Locally approved opportunities" list to `GET` from the admin listing
   endpoint.
10. Decide and implement the `localStorage` dev-fallback question above.
11. Manual test pass mirroring the v0.7 checklist, run against the real
    database (extract → edit → approve → appears on `/route-planning` →
    opens `/opportunity/[id]` → Plant-Based Cooking Class still works →
    non-live rows never appear on public routes).
12. `npm run lint`, `npx tsc --noEmit`, `npm run build`, then commit.

## Risks / open questions

- **RLS correctness is the main risk of choosing Supabase.** A misconfigured
  policy (or RLS left disabled) would expose `draft`/`needs_review` rows
  publicly. Mitigation: write the policy as part of the same migration that
  creates the table (never a separate, easy-to-forget step), and add the
  application-layer `status = 'live'` filter as defense in depth even
  though RLS should already block it.
- **Admin secret management across environments** — local dev, Vercel
  Preview, and Production need separate `ADMIN_TOKEN` values, and this needs
  to be documented so a Preview-deployed admin page isn't left on a
  guessable default.
- **ID stability on re-approval.** v0.7's admin page reuses the same
  generated `id` across repeated "Approve for map" clicks for one
  extraction session to avoid duplicates. The `POST /api/opportunities`
  upsert must preserve this via `ON CONFLICT (id) DO UPDATE`, not a plain
  insert, or re-approving after an edit would create duplicate rows.
- **Soft-delete vs hard-delete for rejected opportunities.** This plan
  proposes keeping rejected drafts in the table with `status = 'rejected'`
  (preserves extraction history, avoids blindly re-extracting the same URL
  later) rather than deleting the row. Worth confirming before building.
- **No admin identity yet** means `admin_notes`/`reviewed_at` have no
  "reviewed by" — acceptable for a single shared admin secret, revisit once
  there's more than one admin.
- **Rate limiting / abuse protection on public `GET` routes** is not
  addressed in this plan — low risk at current scale and traffic, but worth
  a pass before any public launch beyond friends-and-family testing.
- **Whether to build `PATCH /api/opportunities/[id]` in v0.8 at all**, given
  the upsert-by-`id` path already covers edits — current lean is to defer
  it, confirm at implementation time if a lighter-weight status-only update
  turns out to be needed sooner than expected.
